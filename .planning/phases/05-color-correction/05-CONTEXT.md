# Phase 05 Context: 专业调色与光影实现 (Color Correction)

## Domain
实现基础调色、色彩修正、调整图层以及填充描边的 API。用户可以通过大模型对图层进行像素级调色或创建无损调整图层，并能控制前背景色以及选区填充等效果。

## Decisions

### 1. 默认调色策略与人机协同交互
- **前置确认**：当接收到模糊的调色或修图指令时，大模型**绝不擅自决策**，必须通过对话向用户澄清是要使用“破坏性直接调整”还是新建“无损调整图层”。
- **蒙版联动提示**：如果用户选择新建“无损调整图层”且画面当前存有选区，大模型必须额外询问用户是否需将当前选区一并转为新调整图层的蒙版。

### 2. 前/背景色与颜色填充的多模态支持
- **统一颜色格式**：后端调色/填充 API（如 `set_foreground_background_color`）只接收严谨的 `RGB`、`Hex`、`HSB` 以及 `CMYK` 参数值，不处理任何中文颜色短语。
- **视觉提色能力**：遇到无法直接量化的色彩需求时，大模型需主动引导用户上传参考图片，由大模型前端调用自身的视觉能力，提取图片主色调的 Hex 值再下发给后端。

### 3. 高级调色（色阶/色彩平衡）的通道支持
- **专业与求稳并重**：色阶与色彩平衡相关的后端 API 需开放细分通道（红/绿/蓝）的全量参数支持。但在前端的 Schema 参数定义中，应将细分通道标记为可选（Optional）。大模型默认只对主通道（Master）施加影响以确保稳定；仅当用户显式提出“修复偏色”时，再自行计算组合子通道色差。

### 4. 内容识别填充 (CAF) 的无选区防御
- **严格拦截模式**：严禁在无活动选区时私自猜测并使用图层透明度作为选区执行重型的内容识别填充。
- **引导闭环**：若大模型或底层截获到“当前无选区”状态，必须直接中断执行并抛出错误，通过前端提醒用户：“必须框选需要消除的物体才能进行内容识别。请手动框选或让我利用 AI 选择主体后重试”。
